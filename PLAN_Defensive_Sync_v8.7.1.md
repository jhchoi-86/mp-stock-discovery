# 통합 자동 동기화 안정성 강화 작업계획서

기준 버전: v8.7.0 (블루팀 최적화 완료본)
작업 유형: 레드팀 지적사항 기반 방어 로직 보완
작성일: 2026-04-07
작성자: 데니얼 / MetaPrompt Studio
우선순위: 🔴 HIGH — 상용 운영 환경 배포 전 필수 처리

## 1. 배경 및 목적
v8.7.0 블루팀 최적화 작업을 통해 350종목 통합 동기화의 KIS API 호출 횟수가 75% 이상 감소하고, 네이버 수급 데이터 파싱 오류(safeParse)가 해결되었다. 그러나 레드팀 검증 결과 상용 운영 시 장애로 이어질 수 있는 4가지 위험 요소가 식별되었다.
본 작업계획서는 해당 위험 요소를 체계적으로 제거하여 데이터 손실 Zero, 외부 API 차단 Zero, 토큰 만료 무중단을 달성하는 것을 목표로 한다.

## 2. 위험 요소 요약 및 우선순위
| ID | 위험 유형 | 발생 가능성 | 영향도 | 우선순위 |
|---|---|---|---|---|
| R-01 | All-or-Nothing 영속화 → 데이터 전량 손실 | 중 | 🔴 치명 | P1 |

| R-04 | 수급 포맷 함수 중복 정의 → 유지보수 부채 | 저 | 🟡 낮음 | P3 |

## 3. 세부 작업 명세

### ✅ TASK-01 | Checkpoint 중간 저장 로직 구현
- **대상 파일**: server.cjs
- **우선순위**: P1
- **문제 정의**: 현재 동기화 루프는 350개 종목 분석이 전부 완료된 후 signals.json 및 DB에 일괄 저장한다. 349번째 종목 처리 중 서버 재시작 또는 OOM이 발생하면 앞선 348개 분석 결과가 전량 소실된다.
- **구현 명세**:
    - `CHECKPOINT_INTERVAL = 50` 설정
    - 50종목마다 `data/checkpoint_N.json` 저장
    - 이전 체크포인트 자동 삭제 및 최종 저장 후 정리

### ✅ TASK-02 | 네이버 Fallback Throttling 적용
- **대상 파일**: server.cjs
- **우선순위**: P1
- **문제 정의**: KIS API 429 오류 발생 시, 5개 종목 병렬 배치가 동시에 네이버 Fallback을 시도하여 IP 차단 위험이 발생한다.
- **구현 명세**:
    - `fetchNaverWithThrottle(stockCode, index)` 구현
    - 배치 내 순서(index)에 따라 150ms 간격 시차 부여

### ✅ TASK-03 | KIS 토큰 만료 즉시 재발급 로직 복원
- **대상 파일**: server.cjs
- **우선순위**: P2
- **문제 정의**: 최적화 과정에서 토큰 만료(401) 시 재시도 로직이 통합되어 데이터 누락 가능성 발생.
- **구현 명세**:
    - `MAX_TOKEN_RETRY = 1` 적용
    - 토큰 만료 감지 시 즉시 리프레시 후 해당 호출 재시도

### ✅ TASK-04 | 수급 포맷 함수 중복 제거 (리팩토링)
- **대상 파일**: server.cjs, src/utils/supplyRepair.cjs
- **우선순위**: P3
- **문제 정의**: `formatSyncSupply`와 `formatSupply` 중복 존재.
- **구현 명세**:
    - `server.cjs` 내 `formatSyncSupply` 제거 후 공용 함수로 교체
