# 🛡️ MP Stock DB 데이터베이스 무결성 안전 수칙 (DB Safety Protocol)

본 문서는 실서버 운영 중 발생할 수 있는 데이터베이스 누락, 삭제, 다운타임(Downtime) 오류를 원천 차단하기 위한 **Red Team/Blue Team 공통 필수 준수 가이드라인**입니다. 향후 AI 엔지니어 및 개발팀이 DB 스키마를 갱신할 때 무조건 준수해야 하는 규칙입니다.

---

## 1. 실서버(PRD) 테이블 제어 시 `db push` 및 `--accept-data-loss` 사용 절대 금지
- **원인:** `npx prisma db push` 명령어는 개발 환경의 스키마를 DB에 강제로 맞추는 명령어로, 제약 조건(Not Null) 충돌 시 행(Row) 데이터를 통째로 날려버립니다. 오늘 관리자 계정 삭제 사태의 핵심 원인입니다.
- **대책:** 프로덕션 서버에서는 반드시 **`npx prisma migrate deploy`** 를 사용해야 합니다. 
- 스키마가 변경되었을 때는 로컬에서 `npx prisma migrate dev --name <이름>` 으로 SQL 마이그레이션 변경분 파일을 생성한 뒤, 이 안전한 쿼리 파일만 실서버에 적용해야 데이터가 보존됩니다.

## 2. 신규 컬럼(Column) 추가 시 엄격한 빈틈 방어 (Not Null 정책)
- **원인:** 기존에 레코드가 존재하는 상황에서 `String` 처럼 필수값을 강제하는 컬럼을 추가하면, 기존 데이터들은 새 컬럼이 비어있으므로 DB 갱신과 동시에 데이터가 깨지거나 강제 삭제됩니다.
- **대책:** 기존 테이블에 컬럼을 추가할 때는 반드시 다음 중 하나를 따릅니다.
  1. `String?` 처럼 Optional (Null 허용) 타입으로 지정합니다.
  2. `@default("기본값")` 을 지정하여 과거 데이터들도 빈칸이 기본값으로 꽉 채워지도록 유도합니다.

## 3. Enum (열거형) 변경 및 타입 변경 전 필수 백업(pg_dump)
- **원인:** `Role` (PAID -> PRO_USER 등) 값 변경이나 Integer -> String 등의 타입 변경은 Prisma가 데이터를 캐스팅하지 못할 경우 해당 컬럼이나 테이블 전체를 재생성(Re-create)해버려 데이터가 증발합니다.
- **대책:** Enum 및 Type 변형이 감지되는 작업 전에는 반드시 AWS RDS 콘솔에서 수동 스냅샷을 생성하거나 DB 덤프를 백업한 후 스크립트를 적용해야 합니다.

## 4. 백그라운드 무중단 갱신 스크립트화 (배포 쉘스크립트 자동화)
- **대책:** 개발자나 AI가 수동으로 `npx prisma ...` 명령어를 칠 때 휴먼 에러가 발생할 확률이 압도적으로 높습니다. `package.json` 등의 `deploy` 단상승 스크립트에 "안전 확인(Safe mode)" 플래그가 들어간 마이그레이션만 허용하도록 잠금쇠를 걸어두어야 합니다.

## 5. 단계별 검증 절차 (Red Team 체크리스트)
- 개발 모듈 수정 착수 -> **[DB 스키마 영향도 파악]**
- 테이블 명세 변경 발생 시 -> **[Dry-run (가상 실행) 및 경고 문구 확인, 영향 받는 Row 수 카운팅]**
- 안전 보장 시 -> **[마이그레이션 생성 후 SQL문 리뷰]** -> **[실서버 Deploy]**
