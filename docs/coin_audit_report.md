# 🟥 Red Team Audit Report: Coin Discovery System

## 1. 개요 (Audit Overview)
본 문서는 `docs/coin_api_plan.md`에 정의된 코인 종목 발굴 시스템의 기술적 결함, 운영 리스크 및 보안 취약점을 레드팀 관점에서 분석한 결과입니다.

## 2. 식별된 기술적 리스크 (Technical Risks)

### ⚠️ 리스크 1: WebSocket 연결 불안정성 및 상호 배타성
- **분석**: 업비트와 빗썸의 WebSocket은 네트워크 환경에 따라 잦은 연결 끊김(Re-connection)이 발생합니다.
- **결함**: 현재 기획서에는 '연결 유지 및 재시도 로직'에 대한 구체적인 명세가 누락되어 있습니다.
- **조치 권고**: `Exponential Backoff` 기반의 재연결 로직 및 연결 상태 모니터링 기능을 필수 항목으로 추가해야 합니다.

### ⚠️ 리스크 2: 데이터 정규화(Normalization) 부재
- **분석**: 거래소마다 티커 명칭(KRW-BTC vs BTC_KRW)과 데이터 구조가 상이합니다.
- **결함**: 서로 다른 소스의 데이터를 하나의 '추천 점수(Scoring)'로 통합할 때 기준점이 흔들릴 수 있습니다.
- **조치 권고**: 내부 데이터 모델을 하나로 통일하는 `Normalization Layer`를 설계에 포함해야 합니다.

### ⚠️ 리스크 3: AI 정보의 신뢰성 및 뉴스 노이즈
- **분석**: 코인 시장은 주식에 비해 '스팸성 뉴스'와 '가짜 호재'가 매우 많습니다.
- **결함**: 일반 구글 뉴스 RSS만 사용할 경우 AI가 잘못된 정보를 바탕으로 종목을 추천할 위험(Hallucination)이 큽니다.
- **조치 권고**: 거래소 공식 공지사항(RSS/API) 및 공식 트위터 등 '1차 소스'의 가중치를 높이는 필터링 로직을 설계에 추가해야 합니다.

## 3. 운영 및 보안 점검 (Ops & Security)

- **API 권한**: 모든 코인 API 키는 '조회(Read-Only)' 권한으로 한정해야 함을 명시해야 합니다. (자산 탈취 리스크 방지)
- **자원 소모**: 24/7 구동되는 시스템 특성상 메모리 누수(Memory Leak) 발생 시 서비스 전체가 중단될 수 있습니다. `pm2-logrotate`와 더불어 주당 1회 자동 재시작 로직 또는 힙 덤프 모니터링이 필요합니다.

## 4. 최종 검증 합격 요건 (Success Criteria)
1. [ ] 업비트/빗썸 데이터가 동일한 JSON 포맷으로 변환되는가?

3. [ ] AI 코맨트에 '근거 문서의 출처(URL)'가 반드시 포함되는가?
