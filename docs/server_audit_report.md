# 📊 [Audit] 서버 리소스 부하 진단 및 확장성 검토 보고서

## 1. 진단 결과 요약 (Current State)
원격 서버(`15.134.243.209`)에 대한 실시간 SSH 정밀 진단 결과, 현재 서버는 **t3.micro (RAM 1GB)**급 인프라로 확인되었습니다.

- **총 메모리**: 약 911 MB
- **사용 중인 메모리**: 약 624 MB (62.4%)
- **가용 메모리**: **약 287 MB (매우 부족)**
- **Swap 설정**: 없음 (메모리 부족 시 즉각적인 프로세스 킬 가능성 높음)

## 2. 3개 프로그램(K/U/C) 동시 구동 가능성 (Feasibility)

**[부적격: 추가 확장 시 시스템 붕괴 위험]**
1.  **메모리 충돌**: 현재 `ecosystem.config.cjs`에 설정된 개별 프로세스 재시작 기준 임계치(1G)가 물리 하드웨어의 전체 메모리보다 큽니다. 이는 설정 오류이며, 실제로 확장이 이루어질 경우 OOM(Out of Memory)으로 인해 서버가 즉시 중단될 수 있습니다.
2.  **프로세스 병목**: 나스닥 100 스캐너와 코인 Top 100 엔진을 추가할 경우 약 **2GB~3GB의 추가 RAM**이 필요합니다. 현재의 911MB 시스템으로는 수용이 불가능합니다.
3.  **Vite 빌드 제약**: 이미 `aws_update.bat`에서 로컬 빌드를 수행하고 있는 이유도 서버의 RAM이 부족하여 빌드 프로세스를 감당하지 못하기 때문입니다.

## 3. 권고 조치 및 업그레이드 가이드 (Recommendations)

### [권장안] 인프라 규모 상향 (AWS Instance Upgrade)
- **추천 사양**: **t3.medium (4GB RAM)** 또는 **t3.large (8GB RAM)**
- **기기 선정 이유**: 
    - 3개 시장(K-Stock, U-Stock, Crypto)의 실시간 WebSocket 수집 및 데이터 처리 안정성 확보.
    - Python AI 서비스(Gemini 처리용)가 800MB 이상의 힙 메모리를 사용하므로 최소 4GB 이상이 필수적입니다.

### [임시안] 리소스 시분할 운영 (Time-Slicing)
인프라 상향이 어려울 경우, 특정 시간에만 프로세스를 기동하는 방식입니다 (비권장).
- **오전 (09:00-15:30)**: 한국 주식 엔진만 기동.
- **오후 (22:30-05:00)**: 미국 주식 엔진만 기동.
- **상시**: 코인/AI 서비스를 최소 전력으로 기동 (실시간성 감소).

## 4. 결론 (Conclusion)
현재의 `t3.micro` 서버는 글로벌 통합 플랫폼을 성공적으로 운영하기에는 **임계치를 이미 초과**한 상태입니다. 글로벌 알파 발굴의 안정성을 위해 **인프라 확장을 최우선적으로 검토**해 주실 것을 레드팀 및 SRE 관점에서 강력히 건의드립니다.
