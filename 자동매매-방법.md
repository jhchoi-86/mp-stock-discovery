주식 자동매매 프로그램은 크게 3단계로 만들면 됩니다: 1) 증권사 API 연결, 2) 매매전략 구현, 3) 주문·리스크 관리와 모니터링입니다. 한국투자증권 OpenAPI나 키움 OpenAPI를 이용한 국내 주식 자동매매 예제와 프로젝트 구조가 이미 널리 공유되고 있고, 실시간 시세·주문·체결 로그·백테스트까지 포함하는 형태로 구현하는 사례가 확인됩니다.

만들기 전 결정
먼저 어떤 증권사를 쓸지 정해야 합니다. 한국투자증권은 OpenAPI와 웹소켓 기반 실시간 시세 연동 예제가 있고, 키움증권은 자동매매와 백테스트를 포함한 오픈소스·강의 자료가 비교적 많습니다.

또 하나 중요한 점은, 자동매매 프로그램을 외부에 제공하거나 회원에게 종목 추천·신호를 유료로 제공하는 모델은 규제 이슈가 생길 수 있다는 점입니다. 불특정 다수 대상의 자동매매·투자조언은 유사투자자문업/투자자문업/투자일임업 범위와 충돌할 수 있고, 자동매매 프로그램 연동이 자문업 범위로 해석되거나 미등록 투자일임업 문제가 될 수 있습니다.

기본 구조
가장 현실적인 구조는 아래처럼 나누는 것입니다.

인증 모듈: 토큰 발급, 계좌 정보 조회.

시세 모듈: 현재가, 호가, 체결, 캔들 데이터 수집.

전략 모듈: 진입·청산 조건 계산.

주문 모듈: 지정가/시장가/분할매수·분할매도.

리스크 모듈: 1회 손실 한도, 일 손실 한도, 최대 보유 종목 수.

로그/알림 모듈: 체결, 오류, 상태 저장 및 텔레그램/카톡 알림.

이 구조는 한국투자증권 API 예제와 키움 자동매매 프로젝트에서 공통적으로 보이는 구성과도 맞습니다.

추천 개발 순서
모의투자로 먼저 연결합니다.

현재가 조회와 주문 1건만 성공시키는 최소 기능부터 만듭니다.

이동평균선, 거래량, RSI 같은 단순 조건식 전략을 넣습니다.

손절·익절·재진입 제한을 추가합니다.

백테스트와 실거래를 분리합니다.

실거래 전에는 소액으로 검증합니다.

이 순서가 중요한 이유는, 자동매매는 전략보다도 “예외 처리”와 “주문 실패 대응”에서 사고가 많이 나기 때문입니다.

기술 스택
파이썬으로 시작하는 게 가장 무난합니다. 이유는 증권사 API 예제와 오픈소스 자동매매 자료가 파이썬 중심으로 많이 쌓여 있고, 데이터 처리와 백테스트 라이브러리도 연결하기 쉽기 때문입니다.

실시간 데이터는 웹소켓, 주문/조회는 REST API, 저장은 SQLite나 PostgreSQL, 모니터링은 로그 파일 + 알림 봇 조합이 실용적입니다.
서버는 로컬 PC보다 VPS나 항상 켜진 윈도우 서버가 안정적입니다.

최소 예시

규제와 운영
개인용 자동매매는 보통 기술 프로젝트로 시작할 수 있지만, 타인에게 서비스화하면 이야기가 달라집니다. 공개적인 유료 회원제, 종목 추천, 자동 신호 제공, 1:1 조언, 자동매매 프로그램 판매는 자본시장법상 업역 해석이 걸릴 수 있어 사전에 법적 구조를 잡아야 합니다.

즉, “내 계좌에서 쓰는 자동매매”와 “남에게 제공하는 자동매매/신호 서비스”는 완전히 다르게 봐야 합니다.

최소 코드
아래 코드는 토큰 발급, 계좌 잔고 조회, 현재가 조회까지 한 번에 확인하는 기본 뼈대입니다. 엔드포인트의 tr_id는 실전/모의에서 다를 수 있으니 실행 전 KIS 문서에서 맞춰야 합니다.

python
import json
import requests

APP_KEY = "YOUR_APP_KEY"
APP_SECRET = "YOUR_APP_SECRET"
ACCOUNT = "12345678"
PRODUCT = "01"  # 보통 증권계좌 상품코드
BASE_URL = "https://openapi.koreainvestment.com:9443"  # 실전
# BASE_URL = "https://openapivts.koreainvestment.com:29443"  # 모의

def get_access_token():
    url = f"{BASE_URL}/oauth2/tokenP"
    headers = {"content-type": "application/json"}
    body = {
        "grant_type": "client_credentials",
        "appkey": APP_KEY,
        "appsecret": APP_SECRET
    }
    res = requests.post(url, headers=headers, data=json.dumps(body), timeout=10)
    res.raise_for_status()
    return res.json()["access_token"]

def get_price(token, symbol):
    url = f"{BASE_URL}/uapi/domestic-stock/v1/quotations/inquire-price"
    headers = {
        "content-type": "application/json",
        "authorization": f"Bearer {token}",
        "appkey": APP_KEY,
        "appsecret": APP_SECRET,
        "tr_id": "FHKST01010100"
    }
    params = {
        "fid_cond_mrkt_div_code": "J",
        "fid_input_iscd": symbol
    }
    res = requests.get(url, headers=headers, params=params, timeout=10)
    res.raise_for_status()
    return res.json()

def get_balance(token):
    url = f"{BASE_URL}/uapi/domestic-stock/v1/trading/inquire-balance"
    headers = {
        "content-type": "application/json",
        "authorization": f"Bearer {token}",
        "appkey": APP_KEY,
        "appsecret": APP_SECRET,
        "tr_id": "TTTC8434R"
    }
    params = {
        "CANO": ACCOUNT,
        "ACNT_PRDT_CD": PRODUCT,
        "AFHR_FLPR_YN": "N",
        "OFL_YN": "",
        "INQR_DVSN": "02",
        "UNPR_DVSN": "01",
        "FUND_STTL_ICLD_YN": "N",
        "FNCG_AMT_AUTO_RDPT_YN": "N",
        "PRCS_DVSN": "00"
    }
    res = requests.get(url, headers=headers, params=params, timeout=10)
    res.raise_for_status()
    return res.json()

if __name__ == "__main__":
    token = get_access_token()
    print("TOKEN:", token[:20] + "...")

    print("BALANCE:")
    print(json.dumps(get_balance(token), ensure_ascii=False, indent=2))

    print("PRICE:")
    print(json.dumps(get_price(token, "005930"), ensure_ascii=False, indent=2))
바로 다음 확장
이 코드가 되면 그다음은 주문 함수만 추가하면 됩니다. 한국투자증권 API는 주문, 잔고, 현재가, 실시간 체결가까지 API 카테고리가 분리돼 있어 최소 동작 후 단계적으로 확장하기 좋습니다.

실시간 체결
실시간 체결은 REST가 아니라 웹소켓을 씁니다. 국내주식 실시간체결가(H0STCNT0)는 웹소켓 실시간 시세 항목으로 제공되고, 접속키는 oauth2/Approval로 발급합니다.

즉, 최소 동작은 “REST 토큰 → 현재가 조회 → 주문 → 웹소켓 구독” 순서로 붙이면 됩니다.

주의할 점
실전/모의 도메인과 TR ID가 헷갈리면 에러가 자주 납니다. 또한 주문 API는 계좌번호, 상품코드, 해시키(hashkey) 요구 여부가 붙는 경우가 있어 주문 단계에선 문서대로 헤더를 맞춰야 합니다.
